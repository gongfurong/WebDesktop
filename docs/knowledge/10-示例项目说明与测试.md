# 示例项目说明与测试

## 目标

这份文档专门说明示例项目的角色、阅读方式、测试方式和使用边界，帮助：

- 人类读者快速理解示例项目能做什么
- AI 在需要时正确参考示例代码
- 避免把示例项目误当成唯一实现模板或最终产品结构

## 示例项目的定位

示例项目的定位是：

- 参考实现
- 演示实现
- 测试与验证用样例

它的价值在于：

- 提供一个可运行的真实代码样本
- 展示关键技术点如何组合在一起
- 让人类和 AI 都能看到“知识文档中的概念如何落到实际代码上”

它不等于：

- 所有项目的默认最终结构
- 每次任务都必须完整照搬的模板
- 所有平台和所有浏览器的唯一最佳实现

## 人类应该怎么阅读示例

推荐顺序：

1. 先看示例目录下的 `README.md`
2. 再看页面入口文件
3. 再看前端主逻辑
4. 最后看服务端与脚本

如果按典型 Web 示例理解，通常可按下面顺序：

1. `README.md`
2. `index.html`
3. `app.js`
4. `styles.css`
5. `manifest.json`
6. `sw.js`
7. `server.js`
8. `scripts/` 中与该示例配套的脚本

## 人类看示例时应重点关注什么

### 页面入口

看：

- 页面结构如何组织
- 安装引导区域放在哪里
- 关键按钮与状态区如何布局

### 前端主逻辑

看：

- 平台和浏览器识别逻辑
- 安装按钮如何切换
- 主路径、备选路径、不推荐路径如何表达
- 调试信息如何暴露

### 服务端

看：

- 本地静态服务如何提供
- HTTPS 如何启用
- 证书下载或描述文件入口如何提供

### 脚本

看：

- 本地证书如何生成
- 各平台如何导入或删除信任

## AI 应该什么时候参考示例代码

AI 在这些场景中适合参考示例：

- 用户要求“给我一个可运行例子”
- 用户要求“按现有示例扩展”
- 用户要求“把某个知识点落成真实代码”
- 用户要求“验证某个安装或信任流程在代码里怎么实现”

AI 在这些场景中不应默认完整照搬示例：

- 用户只要求单平台最小实现
- 用户只要求解释方案，不要求写代码
- 用户只要求某个局部能力，而示例实现覆盖过多分支

## AI 参考示例时的正确方式

### 1. 先判断任务范围

先判断：

- 当前任务是单平台还是多平台
- 当前任务是局部改动还是完整实现
- 当前任务是示例增强还是正式产品落地

### 2. 只读取必要文件

不要默认把整个示例目录完整读完。

应优先读取：

- 和当前任务直接相关的入口文件
- 和当前平台直接相关的分支逻辑
- 和当前问题直接相关的脚本或服务端代码

### 3. 不要把示例中的所有分支都带入新任务

示例可能包含：

- 多平台分支
- 多浏览器引导
- 调试面板
- 证书工作流

新任务如果没要求这些内容，就不应默认全部搬进去。

### 4. 把示例当作“实现素材”，不是“执行指令”

AI 应该从示例中提取：

- 结构模式
- 核心逻辑
- 关键 API 用法
- 平台差异的处理思路

而不是机械复制整个示例实现。

## 示例项目与知识文档的关系

可以把它们理解成：

- 知识文档：告诉你“为什么”和“有哪些规则”
- 示例项目：告诉你“其中一些规则可以怎样写成代码”

因此：

- 先有知识文档
- 再看示例代码
- 先理解规则
- 再参考实现

## 示例项目与正式项目的关系

示例项目通常更强调：

- 可观察
- 可测试
- 可演示

正式项目通常还要额外考虑：

- 代码分层
- 长期维护
- 安全策略
- 团队协作约束
- 发布流程

因此，示例适合拿来：

- 起步
- 对照
- 参考

但不应被自动视为正式项目的唯一标准答案。

## 如何启动与停止示例项目

### 启动服务

进入示例目录后启动：

```bash
node server.js
```

如果需要本地 HTTPS，先在仓库根目录生成证书，再回到示例目录启动服务。

### 停止服务

如果服务是在当前终端前台运行，通常直接在该终端中按 `Ctrl + C` 即可停止。

如果服务是通过其他方式后台启动的，则应按你当前系统和终端环境的常规方式结束对应进程。

## 如何访问示例项目

### 本机访问

启动后，优先使用终端输出的本地地址访问。

### 非本机访问

如果要让其他设备访问：

1. 使用终端输出的网络访问地址
2. 确认网络可达
3. 确认防火墙允许当前端口
4. 如果要验证更完整安装能力，尽量使用可信 HTTPS

### 访问后先看什么

打开页面后，优先观察：

- 平台识别是否正确
- 浏览器识别是否正确
- 安装支持等级是否符合当前环境
- 诊断信息是否与实际环境一致

## 如何利用示例项目进行测试

### 测试目标

利用示例项目测试时，重点不是“这个示例本身是否好看”，而是验证：

- 安装入口是否出现
- 不同平台和浏览器分支是否按预期工作
- 安全上下文变化后，页面行为是否符合预期
- 证书与信任流程是否可执行

### 推荐测试顺序

1. 先做本机基础访问测试
2. 再做桌面安装入口测试
3. 再做移动端安装路径测试
4. 最后做本地 HTTPS、证书和非本机访问测试

### 一次完整的最小测试流程

1. 启动示例服务
2. 在本机浏览器打开页面
3. 确认页面和游戏能正常运行
4. 确认诊断面板中的平台、浏览器、协议、安全上下文显示正确
5. 在推荐浏览器中测试安装入口
6. 如需移动端测试，再用目标设备访问同一服务
7. 如需 HTTPS 与证书测试，再生成证书并重新访问
8. 测完后停止服务

### 本机基础访问测试

应验证：

- 页面能正常打开
- 游戏能正常运行
- 安装提示区、诊断区、平台识别区能正常显示
- 调试信息与当前实际环境一致

### 桌面安装测试

应验证：

- 在推荐浏览器中是否出现安装入口
- 安装后是否生成桌面或系统应用入口
- 从独立窗口重新打开后，页面是否识别为已安装状态
- 弱支持浏览器是否能给出正确的“仅浏览 / 不推荐主路径”提示

### 移动端安装测试

#### iOS / iPadOS

应验证：

- 是否能通过 Safari 的“共享 -> 添加到主屏幕”完成安装
- 是否能从主屏幕入口再次打开
- 页面是否正确给出主路径和证书提示

#### Android

应验证：

- 推荐浏览器中是否出现原生安装入口
- 若无原生入口，是否正确引导到浏览器菜单安装
- 主屏幕入口创建后能否正常再次打开

### 非本机访问测试

应验证：

- 其他设备是否能通过网络地址访问示例服务
- 当前页面是否根据地址和安全状态切换到正确提示
- 诊断面板是否正确反映当前协议、浏览器和安装能力状态

### 本地 HTTPS 与证书测试

应验证：

- 证书生成脚本能否生成完整材料
- 当前平台的证书导入脚本或导入流程是否可执行
- 导入信任后，不安全提示是否消失或减少
- 安装入口是否比不可信 HTTPS 场景更稳定

### 测试示例时应注意什么

- 示例包含较多分支，不代表每个项目都必须实现全部分支
- 测试时应先确认你当前是在测“主路径”，还是在测“备选路径”
- 不要把临时调试路径误当成正式发布路径
- 如果你只想验证某个平台，不必把所有平台分支都走一遍

### 如果你看到的还是旧界面

如果已经修改了示例代码，但浏览器页面看起来完全没有变化，甚至在服务停止后仍然可访问，通常说明旧的 Service Worker 和缓存还在生效。

优先做法：

1. 启动当前最新版服务
2. 打开页面右上角 `设置`
3. 点击 `清理本地缓存并刷新`
4. 如仍有问题，再使用浏览器开发者工具执行：
   - Unregister Service Worker
   - Clear site data

## 人类快速测试建议

如果你只想快速判断这个示例是否正常：

1. 启动服务
2. 在本机推荐浏览器中打开
3. 看页面是否正常
4. 看安装入口是否出现
5. 看诊断信息是否和当前环境一致

如果这五步都正常，说明示例的大部分主路径至少已经可用。

## 最终原则

- 人类看示例，目的是理解关键实现点
- AI 看示例，目的是提取当前任务需要的最小实现素材
- 示例是参考实现，不是总规范
- 示例应帮助理解知识文档，而不是替代知识文档
