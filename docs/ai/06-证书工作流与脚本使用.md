# AI 证书工作流与脚本使用

## 目的

这份文档专门告诉 AI：

- 什么时候需要进入证书工作流
- 什么时候不需要
- 如何使用现有脚本
- 如何判断站点证书、根证书与信任步骤
- 如何避免误用或过度实现

## 什么时候要读这份文档

只有在用户提到这些内容时才需要继续读：

- 本地 HTTPS
- 不安全提示
- 证书信任
- 局域网或非本机设备访问
- 安装入口缺失且怀疑与安全上下文有关

如果用户没有提这些内容，不要默认进入证书工作流。

## AI 的基本判断顺序

1. 当前是不是只做本机 `localhost` 调试
2. 当前是不是要让非本机设备访问
3. 当前是否明确需要可信 HTTPS
4. 当前是否需要稳定的安装入口、`service worker` 或离线能力
5. 用户是否只想临时访问页面，而不是建立长期可信本地环境

## 默认策略

### 可以不进入证书工作流的情况

- 只是本机 `localhost` 调试
- 只是临时访问页面
- 用户未要求稳定安装与缓存能力

### 应进入证书工作流的情况

- 用户要让非本机设备通过 HTTPS 访问
- 用户要减少不安全提示
- 用户要验证更完整的安装能力
- 用户要验证 `service worker` 或更稳定的 PWA 行为

## 现有脚本的职责

### `scripts/generate-cert.ps1`

职责：

- 生成本地根证书材料
- 生成站点证书材料
- 为当前示例项目写入证书工作目录
- 在 Windows 上辅助完成根证书导入

适合：

- 生成与更新本地证书材料

### `scripts/manage-cert-windows.ps1`

职责：

- 在 Windows 上检查、移除信任根
- 清理本地生成的证书材料
- 执行“恢复到未配置状态”的重置动作

适合：

- Windows 证书工作流的反向操作与状态检查

### `scripts/install-cert-macos.sh`

职责：

- 在 macOS 上导入、删除或检查根证书信任状态

适合：

- 证书材料已经生成后，再做系统信任操作
- 也支持反向移除与清理生成物

### `scripts/install-cert-linux.sh`

职责：

- 在 Linux 上导入、删除或检查系统 CA
- 在需要时处理 NSS 证书库

适合：

- Linux 本地或联调环境中的信任配置
- 也支持反向移除与清理生成物

## AI 应如何理解这些脚本

- 证书生成脚本负责“生成材料”
- 平台脚本负责“导入、移除、检查、清理与重置”
- 这两类动作不要混为一谈

也就是说：

- 生成证书 != 证书已经被系统信任
- 页面可通过 HTTPS 打开 != 安装能力一定完整可用

## 反向工作流应包含什么

一套完整的证书自动化不应只有正向动作，还应至少支持：

1. `status`
   - 看当前是否已生成
   - 看当前是否已导入
2. `remove`
   - 只移除系统或浏览器信任
3. `clean`
   - 只删除本地生成物
4. `reset`
   - 同时移除信任并清理本地生成物

如果没有这四类能力，证书工作流通常不够完整。

## AI 使用这些脚本时应遵循的原则

### 1. 不要默认执行证书工作流

只有用户明确需要，或问题明显与证书/安全上下文相关时，才进入。

### 2. 不要把生成物当源码

证书目录中的这些内容通常属于生成物：

- 根证书
- 站点证书
- 私钥
- CSR
- 序列号文件
- OpenSSL 配置

AI 应优先保留脚本，而不是把生成物当作核心源码依赖。

### 3. 不要把私钥提交到公开仓库

AI 在处理本地 HTTPS 工作流时，应始终把私钥视为敏感文件。

### 4. 不要把临时本地证书方案当正式发布方案

本地证书工作流适合：

- 本地开发
- 局域网联调
- 非本机设备测试

它不应被包装成面向普通用户的正式发布方案。

## AI 输出时应该包含什么

当任务涉及证书工作流时，AI 输出应尽量包含：

1. 为什么需要证书
2. 现在是否真的需要进入证书流程
3. 需要生成什么
4. 需要在哪个平台导入什么
5. 哪些文件是生成物，不应提交
6. 当前方案只是本地调试方案还是长期方案

## AI 不应该做什么

- 不要在用户没提证书时主动引入整套证书方案
- 不要把 HTTPS 可访问误写成证书已被信任
- 不要把根证书、站点证书、私钥、CSR 混为一谈
- 不要把某个平台的导入路径复制到另一个平台
- 不要把生成物目录里的内容默认提交到仓库
